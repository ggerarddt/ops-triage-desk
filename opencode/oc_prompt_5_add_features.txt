Context:
The app now has triage, tests, SQLite-backed users, login, role-based access, governance, audit, and P1 approval gating. This final step should turn the tool into a more complete enterprise-style internal workflow while staying lightweight and local. Supervisors must be able to review, correct, approve, deny, reclassify, and audit high-priority submissions.

Supervisor admin workflow requirements:
1. Admin dashboard
   - Add or complete GET /admin.
   - Require supervisor role.
   - Show three sections:
     - pending P1 escalations,
     - approved P1 incidents,
     - denied or reclassified P1 incidents.
   - Show counts for each section.
   - For each incident, show incident id, title, severity, team, submitter, created time, and status.
2. Supervisor approval
   - Add POST /admin/approve.
   - Accept form data and JSON.
   - Require incident_id.
   - Accept approver from the logged-in supervisor, not from a free-form user field.
   - Accept optional reason.
   - Approve only pending P1 incidents.
   - Record an audit entry.
   - Return a redirect for form requests and useful JSON errors for API-style requests.
3. Supervisor denial
   - Add POST /admin/deny.
   - Accept form data and JSON.
   - Require incident_id and denial reason.
   - Deny only pending P1 incidents.
   - Preserve original severity.
   - Record denied_by, denied_at, reason, and an audit entry.
4. Supervisor reclassification
   - Add POST /admin/reclassify.
   - Accept form data and JSON.
   - Require incident_id and reclassification reason.
   - Allow reclassification from P1 to P2, P3, or P4.
   - Preserve original severity.
   - Record reclassified_to, approver, reason, and an audit entry.
5. Submission editing
   - Add the ability for supervisors to edit pending P1 submissions before approval, denial, or reclassification.
   - Add GET /admin/edit/<incident_id> to render an edit form.
   - Add POST /admin/edit/<incident_id> to save edits.
   - Editable fields:
     - title
     - description
     - business area
     - system affected
     - impact
     - urgency
     - customer impact
   - Re-run severity classification and team routing after edits.
   - If the edited incident is no longer P1, move it to reclassified status and record the new severity.
   - If it remains P1, keep it pending.
   - Do not allow editing approved, denied, or already-reclassified incidents.
   - Record an audit entry that indicates the submission was edited and includes the resulting severity/team.
6. Store design
   - Keep persistence in SQLite.
   - Add incident-store methods such as:
     - get_by_id()
     - get_pending_p1s()
     - get_all_filtered_by(status)
     - approve()
     - deny()
     - reclassify()
     - update_pending()
   - Keep filtering inside the store layer rather than repeating list scans in route handlers.
