Build a small local web application called Operational Incident Triage Desk.

Context:
Keep the application intentionally small, easy to explain, and fully runnable. The goal is to start with a useful working slice that already has the minimum persistence and user model needed for the later supervisor approval workflow.

Use:
- Python
- Flask
- Plain HTML/CSS templates
- SQLite through the Python standard library sqlite3 module
- Werkzeug password hashing
- No external services

Goal:
Create a simple internal tool that lets authenticated operations staff submit an incident report and receive:
1. a severity classification,
2. a suggested owning team,
3. a recommended escalation path,
4. a structured handoff summary,
5. a mock outbound JSON payload for a ticketing or workflow system.

Database and users:
- Add a local SQLite database file, created automatically if missing.
- Add a small schema and seed data.
- Pre-populate exactly these demo users with default passwords:
  - operator1 / ChangeMe123! / role operator
  - operator2 / ChangeMe123! / role operator
  - supervisor1 / ChangeMe123! / role supervisor
- Store hashed passwords, not plaintext passwords.
- Add login and logout.
- Keep sessions simple and local.
- Require login before triage, audit, or admin pages.

Functional requirements:
- A login page.
- A triage web form with these fields:
  - title
  - description
  - business area
  - system affected
  - impact level: low, medium, high
  - urgency: low, medium, high
  - customer impact: yes, no
- A rules-based severity function that classifies incidents as P1, P2, P3, or P4:
  - P1: high impact or high urgency with customer impact yes
  - P2: high impact without customer impact, or medium impact with high urgency
  - P3: medium impact or medium urgency
  - P4: everything else
- A keyword-based routing function that maps incidents to one of:
  - API Integration Team
  - Data Platform Team
  - Customer Operations Team
  - Infrastructure Team
- A result view that shows:
  - severity
  - severity description
  - suggested owner team
  - escalation recommendation
  - structured handoff summary
  - mock outbound JSON payload
- Include three one-click example incident buttons on the triage page:
  - Public Administration: benefits API returning intermittent 502 errors
  - Postal Service: mail sorting ETL pipeline stalled
  - National Research Center: data warehouse nearing disk capacity
- The example buttons must populate the form fields when clicked in Firefox and Chromium.

Role requirements:
- Operators can access only:
  - triage
  - audit
- Supervisors can access:
  - triage
  - audit
  - admin
- The first admin page may be a small placeholder in this prompt, but the route and navigation must exist for supervisors only.

- Use clear function names and simple control flow.
- Add comments only where they help explain business rules.
- Use Jinja rendering safely; do not manually concatenate HTML.
- Include local run instructions.
